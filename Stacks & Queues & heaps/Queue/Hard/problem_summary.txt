1. SLIDING WINDOW MAXIMUM
   Link: https://leetcode.com/problems/sliding-window-maximum/
   
   Problem Explanation:
   - Find maximum element in each sliding window of size k
   - Window slides from left to right across array
   - For each window position, output the maximum element
   
   Example:
   - Input: nums = [1,3,-1,-3,5,3,6,7], k = 3
   - Windows: [1,3,-1] → 3, [3,-1,-3] → 3, [-1,-3,5] → 5,
             [-3,5,3] → 5, [5,3,6] → 6, [3,6,7] → 7
   - Output: [3, 3, 5, 5, 6, 7]
   
   Key Insight - Monotonic Deque:
   - Use deque to store INDICES (not values)
   - Maintain decreasing order of values
   - Deque front always contains index of current window maximum
   
   Algorithm Steps:
   
   1. Remove Out-of-Window Elements:
      - If front index ≤ i - k, it's outside window
      - Pop from front
   
   2. Remove Smaller Elements (Optimization):
      - While queue not empty and nums[back] ≤ nums[i]:
        * Pop from back (useless elements)
      - Smaller elements appear after larger ones, won't be max
   
   3. Add Current Element:
      - Push current index to back
   
   4. Store Result:
      - When i ≥ k-1 (window complete):
        * Front of deque has maximum index
        * Push nums[dq.front()] to result
   
   Time Complexity: O(n)
   - Each element pushed/popped at most once
   - Despite nested loops, amortized O(n)
   
   Space Complexity: O(k)
   - Deque stores at most k elements
   
   Why Deque Works:
   - Can remove from both ends efficiently
   - Maintains useful elements in decreasing order
   - Front always points to maximum
   - Back efficiently removes smaller elements
   
   Comparison with Alternatives:
   - Naive O(n*k): compare all elements in each window
   - Segment Tree: O(n log n) build, O(log n) query (overkill)
   - Heap: O(n log k) but needs lazy deletion
   - Deque: O(n) optimal solution with clean implementation
   
   Key Observations:
   - We need INDICES not values (for window boundary check)
   - Decreasing order ensures front is maximum
   - Elements smaller than current can never be maximum before it
   - Deque perfectly fits this requirement
   
   Why It's Hard:
   - Non-obvious that we need monotonic deque
   - Requires understanding of:
     * Sliding window technique
     * Monotonic data structures
     * Optimization with preprocessing
   - O(n) solution not intuitive on first approach
